You are a senior Python engineer and business intelligence architect.
Build a complete, production-grade Python application that performs competitive

AUTONOMOUS MODE IS NON-NEGOTIABLE:
Once launched with valid CLI arguments, this application must run to completion
without asking for any human input whatsoever. No prompts. No confirmations.
No "are you sure?" dialogs. No pauses. No interactive menus. It starts, it
runs, it saves, it exits. Every decision the app would otherwise ask a human
about must be resolved automatically using the rules defined in this prompt.
If something is ambiguous, the app makes the safest reasonable assumption,
logs what it decided, and continues. It never blocks waiting for input.

───────────────────────────────────────────────────────────────────────────────
SECTION 1 — WHAT THIS TOOL IS (AND IS NOT)
───────────────────────────────────────────────────────────────────────────────

This is NOT a company profiler.
This is NOT an SEO report generator.
This is NOT a directory scraper.

This IS a competitive intelligence engine that:

  1. Reads a project specification document to understand what a new ecommerce
     platform/product intends to offer.

  2. Scans each company in a provided Excel list — live, using real web access —
     to understand what THEY offer across five dimensions.

  3. Compares each company against the project specs to identify:
     - Features and services they have that the project is MISSING
     - UX/design patterns that work well and should be adopted
     - Content strategies and messaging approaches worth borrowing
     - Business model mechanics that could add value to the project
     - Audience segments being served that the project hasn't addressed
     - Gaps in the market that NONE of them are serving well
     - Opportunities: things the project could do BETTER than incumbents

  4. Synthesises all findings into a structured Word document report
     organised as actionable business plan input — not just observations.

The output of this tool feeds directly into a business plan update. Every
finding must be framed as an actionable recommendation, not a description.

SECTION 3 — LIVE WEB SCANNING (PER COMPANY)
───────────────────────────────────────────────────────────────────────────────

For EACH company, perform live scanning using the following sources in order.
Failures in any source must be logged and handled gracefully — never abort.

SOURCE A: FIRECRAWL (primary content extraction)
  Use the Firecrawl API to scrape these pages:
    1. Homepage        ({url}/)
    2. Features/Product ({url}/features OR /product OR /platform OR /solutions)
    3. Pricing         ({url}/pricing OR /plans OR /packages)
    4. About           ({url}/about OR /about-us OR /company)
    5. Customers       ({url}/customers OR /case-studies OR /testimonials)
    6. Blog index      ({url}/blog OR /resources OR /insights — index page only)
    7. Integrations    ({url}/integrations OR /apps OR /marketplace)
  
  Configuration:
    - formats: ["markdown"]
    - onlyMainContent: true
    - waitFor: 1500
    - timeout: 35 seconds per page
    - Max 3500 characters per page (trim to this after extraction)
    - Record which pages returned 404 (signals product completeness)
    - Combine all pages into one structured document with clear PAGE labels
  
  Fallback: if Firecrawl fails entirely for a URL, attempt a direct
  requests.get() with a realistic User-Agent and extract visible text
  using BeautifulSoup as a degraded fallback.

SOURCE B: PLAYWRIGHT (dynamic/JS-heavy pages)
  Use Playwright (headless Chromium) for:
    - Any page that Firecrawl returned empty or near-empty content for
    - The homepage if it appears to be a React/Vue/Angular SPA
    - Any page where JavaScript rendering is needed (detect via empty Firecrawl)
  
  Configuration:
    - wait_until: "networkidle"
    - timeout: 25000ms
    - viewport: {"width": 1280, "height": 800}
    - User-Agent: realistic Chrome on Windows string
    - After load: asyncio.sleep(2) then extract inner_text("body")
    - Max 4000 characters from body text

SOURCE C: COMPANY DATA FROM EXCEL
  Pass the following directly to Claude as context (no scraping needed):
    - Category
    - Niche
    - Description (from Excel)
    - All SEO/traffic columns (DR, traffic, keywords, authority score, etc.)
  
  This is often the most reliable data — always include it regardless of
  whether scraping succeeded.

RATE LIMITING AND CONCURRENCY:
  - MAX_WORKERS = 3 (parallel company processing — configurable via CLI)
  - Delay 0.8 seconds between Firecrawl calls to same API
  - Use asyncio.Semaphore to manage concurrency
  - Exponential backoff: retry up to 3 times (delays: 2s, 4s, 8s)
  - Total timeout per company: 120 seconds — skip and log if exceeded

SECTION 14 — FULL AUTONOMOUS OPERATION PROTOCOL
───────────────────────────────────────────────────────────────────────────────

This section defines every autonomous decision rule the application must follow.
There are NO exceptions. The app must never pause, prompt, or wait for a human.

14.1 — PRE-FLIGHT AUTONOMOUS DECISIONS

  MISSING --specs FILE:
    Auto-action: Scan current directory for any .docx, .pdf, or .txt file.
    If exactly one found → use it, log: "Auto-detected specs file: {filename}"
    If multiple found   → use the most recently modified, log the decision
    If none found       → print clear error, exit with code 1 (this is the
                          only legitimate exit before processing begins)

  MISSING --companies FILE:
    Auto-action: Scan current directory for any .xlsx file.
    If exactly one found → use it, log: "Auto-detected companies file: {filename}"
    If multiple found   → use the most recently modified, log the decision
    If none found       → print clear error, exit with code 1

  OUTPUT FILE ALREADY EXISTS:
    Auto-action: Overwrite silently. Log: "Overwriting existing output: {path}"
    Never ask "are you sure?". Never append a timestamp suffix unless
    --no-overwrite flag is explicitly passed.

  CACHE FILE EXISTS FROM PREVIOUS RUN (without --resume flag):
    Auto-action: Automatically use the cache. Log:
    "Found existing cache ({N} companies). Resuming automatically.
     Pass --fresh to start over."
    This means --resume behaviour is ON BY DEFAULT if a cache file exists.
    The user must explicitly pass --fresh to discard a previous cache.

  MISSING API KEY IN .env:
    Auto-action: Check environment variables directly (os.environ) as fallback.
    If still missing → print which key is absent, exit with code 1.

  UNRECOGNISED COLUMN NAMES IN EXCEL:
    Auto-action: Use fuzzy matching (case-insensitive substring match) to find
    the best column for each required field. Log every mapping decision.
    Example: "Mapped 'Company' → company_name, 'Web URL' → url"
    If URL column genuinely cannot be found → skip that company, log it,
    continue with the rest. Never abort the full run.

  SPECS FILE FORMAT UNRECOGNISED:
    Auto-action: Try all three readers in order: python-docx → pdfminer → plain
    text read. Use whichever succeeds first. Log which reader was used.

14.2 — PER-COMPANY AUTONOMOUS DECISIONS

  FIRECRAWL RETURNS EMPTY FOR A PAGE:
    Auto-action: Mark that page as "empty", try Playwright fallback for that
    specific page only. Do not retry Firecrawl for empty responses (it has
    already returned — it is just empty content, not an error).

  FIRECRAWL RETURNS HTTP ERROR (4xx/5xx):
    Auto-action: Retry up to 3 times with exponential backoff (2s, 4s, 8s).
    After 3 failures → mark page as "failed", continue with other pages.
    Log: "[WARN] Firecrawl failed for {url}/{page} after 3 retries: {error}"

  PLAYWRIGHT TIMES OUT:
    Auto-action: Set a hard 25-second timeout. On timeout → log the failure,
    return whatever partial text was extracted, continue. Never hang.

  ALL SCRAPING FAILS FOR A COMPANY (Firecrawl + Playwright both fail):
    Auto-action: Run Claude analysis on Excel data ONLY (category, niche,
    description, SEO metrics). Mark company as "excel_only" in cache.
    Do not skip the company. Partial analysis is better than no analysis.

  CLAUDE RETURNS MALFORMED JSON:
    Auto-action: Strip markdown fences, attempt json.loads(). On failure,
    try ast.literal_eval(). On second failure, retry Claude once with a
    simplified prompt requesting only 8 core fields. On third failure,
    store raw text in cache under key "raw_claude_response", mark as
    "parse_failed", continue. Never block on JSON errors.

  CLAUDE API RATE LIMIT (429):
    Auto-action: Implement token bucket rate limiting. On 429 response,
    wait exactly 60 seconds, then retry. Log every wait.
    Do not reduce quality or skip analysis — just wait and retry.

  CLAUDE API TIMEOUT OR CONNECTION ERROR:
    Auto-action: Retry up to 3 times with 10-second delays.
    After 3 failures → mark company as "claude_failed", continue.

  COMPANY URL RETURNS 404 OR REDIRECT LOOP:
    Auto-action: Try the apex domain if a path was given. Try http:// if
    https:// fails. After two attempts → mark as "url_unreachable",
    run Claude on Excel data only, continue.

  COMPANY URL IS BLANK OR MALFORMED IN EXCEL:
    Auto-action: Skip scraping entirely. Run Claude on Excel data only.
    Log: "[SKIP] No valid URL for {company_name} — analysis from Excel only"

  DUPLICATE COMPANIES IN EXCEL (same domain appears twice):
    Auto-action: Process the first occurrence. Mark subsequent duplicates
    as "duplicate_skipped" in the cache. Log each duplicate found.

  COMPANY COUNT EXCEEDS 500:
    Auto-action: Process all of them. The 500 figure is approximate —
    never truncate the list unless --limit is explicitly set.

14.3 — SYNTHESIS AND REPORT AUTONOMOUS DECISIONS

  SYNTHESIS CLAUDE CALL EXCEEDS CONTEXT LIMIT:
    Auto-action: Split companies into batches of 100. Run synthesis on each
    batch. Then run a final meta-synthesis across all batch summaries.
    Log: "Synthesis split into {N} batches due to context limits"

  WORD DOCUMENT VALIDATION FAILS:
    Auto-action: Attempt auto-repair (unpack XML, fix known issues, repack).
    Log every repair action taken. If repair fails after 2 attempts →
    save a plain-text fallback report as {output_name}_fallback.txt
    containing all findings in structured plain text. Never exit without
    saving SOMETHING.

  OUTPUT DIRECTORY DOES NOT EXIST:
    Auto-action: Create it with os.makedirs(exist_ok=True). Never fail on
    a missing directory.

  NODE.JS NOT INSTALLED (Word doc builder requires it):
    Auto-action: Detect at startup with shutil.which("node"). If missing →
    fall back to python-docx for the entire Word document. Log the fallback.
    The report is produced regardless of whether Node.js is present.

14.4 — SIGNAL HANDLING AND GRACEFUL SHUTDOWN

  CTRL+C (KeyboardInterrupt):
    Auto-action: Catch the signal. Save the current cache immediately.
    Print: "Interrupted. Progress saved to {cache_file}. Re-run to resume."
    Exit cleanly with code 0. Never lose completed work on interruption.

  SYSTEM CRASH / UNEXPECTED EXCEPTION:
    Auto-action: Wrap the entire main() in a top-level try/except.
    On any unhandled exception → write the cache, write an error report
    to gap_analyzer_crash.log with full traceback, exit with code 1.

  LONG RUN HEARTBEAT:
    Every 5 minutes of processing, write a heartbeat line to the log:
    "[HEARTBEAT] {timestamp} — {N}/{total} companies processed, {M} remaining"
    This ensures the user can monitor progress without interaction.

14.5 — AUTONOMOUS LOGGING PROTOCOL

  All autonomous decisions must be logged. Log format:
    [TIMESTAMP] [LEVEL] [SOURCE] Message

  Levels: INFO, WARN, ERROR, DECISION, HEARTBEAT
  
  DECISION level is used specifically when the app makes an autonomous choice
  that a human might have been asked about:
    [2026-03-19 14:23:01] [DECISION] [PRE-FLIGHT] Auto-detected specs file: project_specs.docx
    [2026-03-19 14:23:01] [DECISION] [PRE-FLIGHT] Cache found (247 companies). Resuming automatically.
    [2026-03-19 14:25:44] [DECISION] [klaviyo.com] Playwright fallback used — Firecrawl returned empty homepage
    [2026-03-19 14:31:02] [DECISION] [gorgias.com] All scraping failed — Claude running on Excel data only

  Log to both console (stdout) and gap_analyzer.log file simultaneously.
  Console shows INFO+ only. Log file captures everything including DEBUG.

SECTION 15 — SESSION AND MEMORY RETENTION PROTOCOL
───────────────────────────────────────────────────────────────────────────────

This section defines how the application preserves, restores, and builds on
its own state — both within a single run and across multiple runs over time.
The goal: the tool gets smarter with every run and never loses work.

15.1 — WITHIN-RUN STATE (the live session)

  STATE OBJECT:
    Maintain a single in-memory AppState object throughout the run:

    AppState {
      run_id          : str   # UUID generated at start, e.g. "run_20260319_142301"
      run_started_at  : str   # ISO timestamp
      specs_file      : str   # resolved absolute path
      companies_file  : str   # resolved absolute path
      output_file     : str   # resolved absolute path
      project_profile : dict  # extracted from specs — the comparison baseline
      total_companies : int
      processed       : int   # incremented after each company completes
      succeeded       : int
      partial         : int   # scrape degraded but analysis completed
      failed          : int
      companies       : dict  # domain → analysis result (grows during run)
      synthesis       : dict  # populated at end of run
      errors          : list  # all errors encountered
      decisions       : list  # all autonomous decisions logged
    }

  WRITE FREQUENCY:
    The AppState is serialised to the cache file (gap_analyzer_cache.json)
    after EVERY single company completes. Not in batches. After every one.
    This ensures that if the process dies after company 347, companies 1-347
    are all saved and the next run resumes from 348.

  ATOMIC WRITES:
    Write cache to a temp file first ({cache}.tmp), then rename to final
    path. This prevents a corrupt cache if the process dies mid-write.

15.2 — CROSS-RUN PERSISTENCE (the long-term memory)

  CACHE FILE STRUCTURE:
  {
    "metadata": {
      "run_id":         "run_20260319_142301",
      "created_at":     "2026-03-19T14:23:01Z",
      "last_updated":   "2026-03-19T16:47:22Z",
      "specs_file":     "project_specs.docx",
      "companies_file": "companies.xlsx",
      "total_companies": 500,
      "runs": [
        {
          "run_id":     "run_20260319_142301",
          "started":    "2026-03-19T14:23:01Z",
          "ended":      "2026-03-19T16:47:22Z",
          "completed":  487,
          "partial":    9,
          "failed":     4
        }
      ]
    },
    "project_profile": { ...extracted from specs... },
    "companies": {
      "klaviyo.com": {
        "status":      "success",           // success / partial / failed
        "run_id":      "run_20260319_142301",
        "analysed_at": "2026-03-19T14:31:55Z",
        "source":      "firecrawl+claude",  // firecrawl+claude / excel_only / playwright+claude
        "pages_scraped": ["homepage", "pricing", "about", "features"],
        "pages_failed":  ["integrations"],
        "analysis":    { ...full JSON from Claude... }
      }
    },
    "synthesis": { ...populated after final synthesis call... },
    "decisions_log": [ ...all DECISION-level log entries... ]
  }

  CACHE MERGING ACROSS RUNS:
    When a new run starts with an existing cache:
    - Load existing cache into AppState
    - For each company in the Excel: check if domain exists in cache
    - If exists AND status is "success" → skip (use cached result)
    - If exists AND status is "partial" or "failed" → RE-ANALYSE
      (partial/failed companies are always retried on resume)
    - If not in cache → analyse fresh
    - After all companies: re-run synthesis with full combined results
    - Append new run metadata to the "runs" array in cache
    This means each run IMPROVES the cache, never overwrites it wholesale.

  SPECS CHANGE DETECTION:
    Hash the specs file content at startup (md5). Store hash in cache metadata.
    On resume: if hash has changed → invalidate project_profile, re-extract,
    log: "[DECISION] Specs file changed — project profile re-extracted"
    Company analyses are NOT invalidated (the specs change only affects how
    findings are framed in the final report, not the raw company data).

  COMPANIES FILE CHANGE DETECTION:
    Hash the companies Excel at startup. Store in cache metadata.
    On resume: if hash has changed →
    - Load new Excel, compare domain list to cache
    - New domains not in cache → add to analysis queue
    - Domains in cache but removed from Excel → keep in cache (never delete)
      but exclude from the current report's company section
    - Log all additions detected

15.4 — INCREMENTAL UPDATES (re-running on new data)

  The tool is designed to be run repeatedly as new companies are added to
  the Excel file or as the project specs evolve. The cache/resume system
  makes this efficient:

  SCENARIO: User adds 50 new companies to the Excel
    → Run the tool again with the same --companies flag
    → Existing 450 companies load from cache instantly
    → Only the 50 new companies are scraped and analysed
    → Synthesis is re-run with all 500 results
    → New report reflects the full 500 including the new additions
    → Total runtime: ~15 minutes instead of 3+ hours

  SCENARIO: User updates the specs document
    → Run the tool again with the same flags
    → All company analyses load from cache
    → Project profile is re-extracted from new specs
    → Synthesis re-run with updated project profile for comparison
    → New report reflects updated strategic recommendations
    → Total runtime: ~5 minutes (synthesis only)

  SCENARIO: User wants to refresh a specific company
    → Pass --refresh "klaviyo.com" (support multiple: --refresh "a.com,b.com")
    → Those specific companies are re-scraped and re-analysed
    → All others load from cache
    → Report regenerated with fresh data for specified companies
